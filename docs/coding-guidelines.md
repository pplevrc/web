# コーディングガイドライン

本プロジェクトにおける、コンポーネントの配置や設計についてのガイドラインです。

### ディレクトリ構成例

上記のルールに基づいたディレクトリ構成の例です。

```text
public/                                    # 静的アセット (ビルド時にコピーされる想定)
public-mock/                               # モックデータ (ビルド時にコピーされる想定)
scripts/                                   # ビルド時に参照されるスクリプト
src/
├── assets/                                # 静的アセット (Vite から参照される想定)
├── pages/
│   └── (path-of-page).astro               # ルーティング/レイアウト/データ取得のみ
├── components/
│   ├── pages/                             # 各ページ固有のUIコンポーネント置き場のルート
│   │   └── (name-of-page)/
│   │       ├── index.astro                # トップページUIエントリーポイント
│   │       └── (component-name)/
│   │           └── (component-name).astro # ページ固有のコンポーネント
│   └── (domain-name)/                     # 汎用またはドメインで共通のコンポーネント
│       └── (component-name)/
|           └── (component-name).astro     # 汎用またはドメインで共通のコンポーネント
├── contents/
│   └── (content-name)/
│       └── index.ts                       # CMS からコンテンツを取得するロジック
├── layouts/
│   └── (layout-name).astro                # レイアウトコンポーネント
├── styles/                                # スタイルシート (PandaCSS から自動生成される)
└── lib/
    └── (library-name).ts                  # 汎用ロジック・ライブラリ

```

## コーディングルール

### 1. ページのエントリーポイントと責務の分離

- **`pages/` 配下**: レイアウト、メタデータ、およびデータフェッチ処理のみを担当します。UIの具体的な構造やスタイリングは行いません。
- **ページのUIメイン部分**: `components/pages/<page-name>/index.astro` などをUIのエントリーポイントとし、`pages/` 側からはデータ（Props）を渡してこのコンポーネントを呼び出す形にしてください。

### 2. コンポーネントの配置場所

- **ページ固有のコンポーネント**: 特定のページ（およびそのバリエーション）でのみ使用されるコンポーネントは、`components/pages/<page-name>/` 配下に配置します。
  - 例：特定のページのみで使われる見た目の異なるコンポーネントやバリアント。
- **汎用コンポーネント**: 複数のページで再利用される汎用的、あるいはドメイン固有のコンポーネントは、`components/` 直下に**ドメイン別**でディレクトリを作成して配置します。
  - 例：`components/ui/`, `components/user/`, `components/feature/` など。

### 3. インポートの制限

- src/* と scripts/* は互いに参照してはならない
